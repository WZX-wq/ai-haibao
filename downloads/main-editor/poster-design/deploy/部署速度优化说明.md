# 部署速度优化说明

当前仓库保留了两条部署路径：

1. 源码构建部署
   入口：`deploy/docker-compose.yaml`
   特点：服务器需要执行镜像构建。

2. 预构建镜像部署
   入口：`deploy/pack-fast-deploy.ps1` 或 `deploy/pack-prebuilt-image-bundle.ps1`
   特点：在本地提前构建镜像，服务器只需要 `docker load` 和 `docker compose up -d`。

## 当前排查结论

实际卡顿点不在 `npm ci`。

- `web` 之前慢，主要是 Docker build context 过大；现在已经通过 `.dockerignore` 缩小。
- `api` 的 `npm ci` 和 `webpack build` 都很快。
- `api` 的完整构建耗时主要集中在运行时镜像安装浏览器：
  `apt-get update && apt-get install ...`

为降低这部分耗时，当前默认已经切换为：

- `CHROMIUM_PACKAGE=chromium-headless-shell`
- `PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium-headless-shell`
- 支持 `APT_HTTP_PROXY` / `APT_HTTPS_PROXY`
- 关闭 apt 语言包索引，减少首次 `apt-get update` 的额外下载

另外新增了浏览器运行时基础镜像拆分：

- `deploy/Dockerfile.api-base`
- `deploy/Dockerfile.api.fast`

`Dockerfile.api-base` 只负责构建一次浏览器运行时层。
`Dockerfile.api.fast` 复用这个基础镜像，避免每次预构建打包都重新安装浏览器。

这更符合当前服务“服务端无头截图”的实际用途。

## 推荐方案

如果目标是“部署最快”，优先使用预构建镜像部署：

```powershell
powershell -ExecutionPolicy Bypass -File deploy\pack-fast-deploy.ps1
```

生成：

- `deploy/poster-design-prebuilt-images-deploy.zip`

服务器上只需要：

```bash
unzip -oq poster-design-prebuilt-images-deploy.zip
cd prebuilt-image-bundle
cp env.template .env
docker load -i images/poster-design-api-prebuilt.tar
docker load -i images/poster-design-web-prebuilt.tar
docker compose up -d
```

## 如果仍然要服务器源码构建

当前源码构建路径已经做了这些优化：

- 前端 build context 大幅缩小
- npm registry 统一为腾讯镜像
- lockfile host 已统一
- API 改为默认安装 `chromium-headless-shell`

但只要服务器仍然现场安装浏览器依赖，完整 API build 依然会明显慢于预构建部署。

## 新的快速打包行为

`deploy/pack-prebuilt-image-bundle.ps1` 现在会：

1. 检查本地是否已有 `poster-design-api-runtime:headless-shell`
2. 如果没有，先构建一次浏览器运行时基础镜像
3. 再使用 `deploy/Dockerfile.api.fast` 构建业务 API 镜像

这意味着：

- 第一次打包仍然可能慢
- 第二次及之后打包会明显更快

如果你想强制重建浏览器基础镜像，可以先设置：

```powershell
$env:FORCE_REBUILD_API_BASE='1'
```

如果你的网络对 Debian 镜像仍然慢，可以在构建前额外设置代理：

```powershell
$env:APT_HTTP_PROXY='http://127.0.0.1:7890'
$env:APT_HTTPS_PROXY='http://127.0.0.1:7890'
```

然后再执行：

```powershell
docker build -f deploy/Dockerfile.api-base -t poster-design-api-runtime:headless-shell .
```
